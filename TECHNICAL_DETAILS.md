# Swarm Technical Details

## Overview
Swarm is an async multi-agent software factory that turns a plain-English request into a generated runnable project under output/{project_name}.

Primary capabilities:
- 6 specialized agents with distinct personalities and roles
- Real-time terminal dashboard using textual
- SQLite-backed message and phase persistence
- OpenAI-compatible model integration (GitHub Models endpoint)
- Deterministic output writing and project bootstrapping

## Tech Stack
- Python 3.11+ target
- asyncio for orchestration and concurrent agent turns
- textual for TUI rendering
- openai AsyncOpenAI client for model calls
- aiosqlite for async persistence
- PyYAML for configuration loading

## Runtime Entry
Main entrypoint:
- main.py

CLI usage:
- python main.py "build me a URL shortener"
- python main.py "build me a URL shortener" --headless

Key responsibilities in main.py:
- Parse CLI args (task, config path, headless mode)
- Load swarm.config.yaml
- Resolve API key from config first, then GITHUB_TOKEN
- Construct MessageBus and BuildEngine
- Run either textual dashboard mode or async headless mode

## Architecture
Core orchestration components:
- core/build_engine.py
- core/message_bus.py
- core/file_writer.py

### BuildEngine
BuildEngine is the central coordinator.

It defines 7 sequential phases:
1. PLANNING
2. ARCHITECTURE
3. IMPLEMENTATION
4. TESTING
5. DOCUMENTATION
6. PACKAGING
7. COMPLETE

Phase behavior:
- Selects participants per phase
- Runs participating agent turns concurrently with asyncio.gather
- Publishes phase events and status updates
- Injects requirement change from PM in ARCHITECTURE or IMPLEMENTATION
- Enforces a realistic testing event where Tester reports at least one issue
- Emits backend fix event before phase completion
- Generates final project scaffold and writes it to output/{project_name}

### MessageBus
MessageBus responsibilities:
- Persist messages into SQLite messages table
- Persist phase progress into SQLite phases table
- Broadcast events to subscribed consumers
- Provide async stream() generator used by dashboard

Persistence schema:
- messages(id, source, target, phase, kind, status, text, timestamp)
- phases(id, phase, progress, total, timestamp)

### FileWriter
FileWriter responsibilities:
- Safe path handling for generated files
- Directory creation
- Async file writes via asyncio.to_thread
- Optional git initialization in generated output

Git steps attempted (if git available):
- git init
- git add .
- git commit -m "Initial scaffold generated by Swarm"

## Agent System
Agent modules:
- agents/base_agent.py
- agents/architect.py
- agents/backend.py
- agents/frontend.py
- agents/tester.py
- agents/docs.py
- agents/pm.py

### BaseAgent
Common behavior:
- Defines system prompt composition
- Calls AsyncOpenAI chat completions when API key is available
- Falls back to deterministic local lines when LLM is unavailable
- Streams per-line updates to MessageBus (live panel behavior)
- Supports reaction messages to other agents

### Personalities and Roles
- Architect: system boundaries, structure-first, opinionated on design
- Backend: pragmatic implementation focus, anti-overengineering
- Frontend: UX-first, strict on predictable contracts
- Tester: skeptical, fault-focused, quality gate behavior
- Docs: implementation-aligned documentation and consistency checks
- PM: scope and requirement perturbation to simulate real-world change

## Dashboard (Textual UI)
UI module:
- ui/dashboard.py

Layout implemented:
- Top bar: SWARM title, task, elapsed time
- Main grid: 6 agent panels with status and live scrolling logs
- Feed panel: inter-agent message stream
- Bottom bar: phase label and progress bar

Rendering behavior:
- Consumes MessageBus stream asynchronously
- Routes agent messages to panel-specific logs
- Mirrors all events into shared feed
- Advances progress on phase events

## Error Handling and Resilience
Implemented safeguards:
- Config file existence checks
- Model call exception fallback to deterministic local agent output
- Subscriber isolation in MessageBus (one failing subscriber does not stop swarm)
- SQLite write guards and lock-protected persistence
- Path traversal prevention in FileWriter
- Output git bootstrap treated as best-effort

## Concurrency Model
Concurrency primitives:
- asyncio tasks and gather for parallel agent execution
- asyncio.Queue for message event buffering
- async generator for dashboard streaming
- lock-guarded DB writes for consistency

Design outcome:
- Live, interleaved updates from multiple agents
- Ordered phase progression with concurrent work inside each phase
